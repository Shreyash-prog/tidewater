# Tests

Cross-cutting and component test suites. Component tests live next to their
concern (`tests/remediator/`, `tests/policy_engine/`, `tests/runbook/`);
repo-wide tests live at this top level. Run everything with `make test`
(excludes the `smoke` marker). Lambda/infra imports resolve via the layout
configured in the repo-root `conftest.py`; pytest runs with
`--import-mode=importlib`.

## Suites

- **`remediator/`** — the remediator Lambda: rule_id → SSM document selection,
  per-rule parameter builders, the protected-role guard, idempotent
  re-invocation, and the flag-only `iam.wildcard_policy` (asserted to never
  dispatch).
- **`runbook/`** — the Phase 5 SSM Automation runbooks. `test_new_runbooks.py`
  asserts each runbook's structure: required parameters, guardrail-before-
  snapshot, **snapshot-before-any-mutation**, `assertNoAttachments` before
  policy deletion, and that `delete_iam_access_key.yml` deactivates
  (`UpdateAccessKey`) but never `DeleteAccessKey`. `test_snapshot_and_audit.py`
  exercises the canonical shared helpers (`runbooks/_shared/`) against moto.
- **`policy_engine/`** — the policy engine Lambda (decisions, approval
  idempotency, re-dispatch).

## `test_iam_grants_match_boto3_calls.py` — the boto3 ↔ IAM grant audit

The IAM detectors call read-only IAM APIs directly via boto3. Their execution
role's permissions are **hand-written** in `infra/stacks/core_stack.py` rather
than generated by a CDK `grant_*` helper, so code and grants can drift apart:
add an `iam.get_*` call without the matching grant and the only signal is an
`AccessDenied` at runtime, in production.

This meta-test statically scans the detector source tree for boto3 IAM method
references — direct calls (`iam.get_role(...)`), bare callable references
(`self._exists(self.iam.get_user, ...)`), and paginator strings
(`get_paginator("list_roles")`) — maps each to its IAM action via
`BOTO3_METHOD_TO_IAM_ACTION`, synthesizes `CoreStack` in-process, and asserts the
detector Lambda's role grants every required action.

**When you add a new IAM call to a detector, make two edits:**

1. Grant the `iam:` action on the detector role in `core_stack.py`.
2. Add the `boto3_method -> "iam:Action"` pair to `BOTO3_METHOD_TO_IAM_ACTION`
   in this test.

The test fails until both are present. A companion check
(`test_method_map_is_complete_for_scanned_trees`) fails if a scanned tree uses an
IAM-looking paginator method that isn't in the map, so the map can't silently
fall behind.

The scan covers, per `HANDLER_TO_LAMBDA_LOGICAL_ID`, the detector source tree
mapped to its synthesized Lambda logical-id prefix. The remediator is **not**
scanned for IAM grants: it never calls IAM directly — all destructive IAM
actions run inside SSM Automation under `TidewaterSsmExecutionRole`.
